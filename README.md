## ⚙️ Архитектура данных и автоматизация (Data Pipeline)

Система сочетает в себе строгий контроль качества при первичной загрузке и полную автоматизацию рутинных обновлений через CI/CD.

### 1. Первичная загрузка (Manual Ingestion)
*   **Сбор:** Извлечение сырых данных ГИБДД в формате JSON через Jupyter Notebook [`data_loading_api.ipynb`](./data_loading_api.ipynb).
*   **Валидация:** Данные проходят локальную проверку на целостность и соответствие схемам.
*   **Буферизация:** Через **GitHub Desktop** файлы загружаются в репозиторий. GitHub используется как основное облачное хранилище и версионный архив — после «пуша» локальные файлы удаляются.

### 2. Синхронизация с БД (GIBDD Sync)
*   **Триггер:** GitHub Action запускается по расписанию (раз в месяц) или при обновлении архива. [`scriptsupdate_data.ipynb`](./scriptsupdate_data.ipynb)
*   **Процесс:** Автоматический скрипт считывает новые JSON-файлы из репозитория и выполняет `UPSERT` данных в **Supabase (PostgreSQL)**, обновляя актуальную базу для дашборда.

### 3. Ежедневные метеообновления (Weather Update)
*   **Источник:** API **Open-Meteo**. Выбрано за высокую точность исторических данных и удобную работу без ключей доступа.
*   **Автоматизация:** Каждое утро в 09:00 (МСК) скрипт собирает актуальные метеоданные (температура, осадки, состояние видимости) по координатам регионов.
*   **Цель:** Обогащение статистики ДТП погодным контекстом для выявления корреляций.

### 4. Справочники и геоданные (City Reference)
*   **Парсинг Wikipedia:** Использование библиотек `Requests` для автоматического сбора актуальных списков городов России, их статуса и численности населения.
*   **Геокодирование:** Очистка данных, привязка кодов ОКАТО через **Dadata API**, а также расчет географических центроидов через **Geopy**.
*   **Регламент:** Актуализация справочников проводится 2 раза в год, что гарантирует точность демографических данных и административных границ.

## 📁 Структура репозитория
*   `data_loading_api.ipynb` — Ноутбук для сбора данных и первичной обработки.
*   `init_db.sql` —  Скрипт инициализации структуры базы данных (DDL), создание таблиц, индексов и связей.
*   `db-schemas` - Схемы базы данных (DDL).
*   `data_analysis.ipynb` — Исследовательский анализ данных (EDA).
*   `dashboard_screenshots` — Скриншоты дашборда.
*   `scriptsupdate_data.ipynb` — Конфигурации автоматических запусков (CI/CD) и Python-скрипты для обновления погоды и синхронизации с базой.

## 📍 География и масштабируемость
На текущем этапе проект реализован на примере данных по трем крупнейшим узлам Сибири и Урала: Омск, Новосибирск и Тюмень.
*    **Масштабируемость:** Архитектура решения универсальна и позволяет быстро подключить любой другой регион РФ.
*    **Аналитика:** Сравнение городов со схожим климатом помогает выявить, где дорожная инфраструктура лучше справляется с погодными аномалиями.

## 💡 Рекомендация по использованию (Insights)
Данные на дашборде позволяют проводить кросс-региональный анализ и выявлять лучшие практики управления безопасностью движения в сложных погодных условиях.
